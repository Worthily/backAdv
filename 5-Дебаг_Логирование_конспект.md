## Введение: дебаг vs логирование

| Понятие | Назначение | Аудитория | Среда |
|---------|------------|-----------|-------|
| **Дебаг (debug)** | Найти и устранить ошибку **сейчас** | Разработчик | Локально, staging |
| **Логирование (logging)** | Зафиксировать событие **для анализа потом** | Разработчик, DevOps, SRE | Dev, staging, **prod** |

```
Разработка                    Production
──────────                    ──────────
var_dump, Xdebug, dd()   →    Monolog, structured logs
console.log              →    Sentry, APM, error tracking
```

> **На экзамене:** дебаг — инструмент **разработки**; логирование — **постоянный** механизм наблюдаемости, особенно на проде.

## 1. Дебаг в PHP

### Что такое дебаг

**Дебаг (отладка)** — процесс поиска причин неверного поведения программы: где упало, какие значения у переменных, какой путь выполнения прошёл код.

### Уровни инструментов (от простого к продвинутому)

| Инструмент | Когда | Плюсы / минусы |
|------------|-------|----------------|
| `echo`, `print_r`, `var_dump` | Быстрая проверка | Просто; ломает вывод, забывают удалить |
| `error_log()` | Вывод в лог без HTML | Не портит страницу |
| `die` / `exit` + dump | Остановка в точке | Грубо, только для локали |
| **Xdebug** | Пошаговая отладка | Breakpoints, watch, stack trace |
| **DD** (`dd`, `dump`) | Laravel/Symfony | Красивый вывод, останавливает выполнение |
| **Profiler** (Xdebug, Blackfire) | Медленный код | Время, память, вызовы |

---

### 1.1 Базовый вывод: var_dump, print_r

```php
$data = ['user' => ['id' => 1, 'name' => 'Anna'], 'items' => [10, 20]];

// Подробно: типы, вложенность
var_dump($data);

// Компактнее для массивов
print_r($data);

// Читаемый вывод в <pre> (только dev!)
echo '<pre>' . htmlspecialchars(print_r($data, true)) . '</pre>';
```

| Функция | Показывает типы | Возвращает строку |
|---------|-----------------|-------------------|
| `var_dump()` | Да | Нет |
| `print_r()` | Нет | Опционально (`true`) |

---

### 1.2 error_log — вывод в лог сервера

Не мешает HTTP-ответу (в отличие от `echo` в середине JSON/HTML).

```php
error_log('Order #' . $orderId . ' created');

// С контекстом (PHP 8+)
error_log(json_encode([
    'event' => 'payment_failed',
    'order_id' => $orderId,
    'error' => $e->getMessage(),
], JSON_UNESCAPED_UNICODE));
```

Путь к логу задаётся в `php.ini`: `error_log = /var/log/php_errors.log`.

---

### 1.3 Настройка отображения ошибок (php.ini / runtime)

```ini
; Локальная разработка
display_errors = On
display_startup_errors = On
error_reporting = E_ALL

; Production
display_errors = Off
log_errors = On
error_reporting = E_ALL & ~E_DEPRECATED & ~E_STRICT
```

```php
// В коде (осторожно на проде!)
ini_set('display_errors', '1');
error_reporting(E_ALL);
```

| Режим | `display_errors` | Куда попадает ошибка |
|-------|------------------|----------------------|
| **Dev** | On | Экран + можно в лог |
| **Prod** | Off | Только в лог / мониторинг |

---

### 1.4 Xdebug — пошаговая отладка

**Xdebug** — расширение PHP: breakpoints, step over/into, просмотр переменных, профилирование.

**Типичный workflow:**

1. Установить Xdebug (`pecl install xdebug` или пакет ОС).
2. Настроить `xdebug.mode=debug` (и `develop` для улучшенного `var_dump`).
3. IDE (PhpStorm, VS Code) слушает порт **9003**.
4. Trigger: cookie `XDEBUG_SESSION=1`, GET-параметр или расширение браузера.

```ini
; php.ini (фрагмент)
zend_extension=xdebug
xdebug.mode=debug,develop
xdebug.start_with_request=trigger
xdebug.client_host=127.0.0.1
xdebug.client_port=9003
```

**Возможности для экзамена:**

- **Breakpoint** — остановка на строке
- **Watch** — отслеживание выражения
- **Stack trace** — цепочка вызовов до ошибки
- **Step Over / Into / Out** — пошаговое выполнение

### 1.5 Profiler — поиск медленного кода

**Profiler** — инструмент, который измеряет, где приложение тратит время и память. Если `var_dump` отвечает на вопрос «что лежит в переменной?», а Xdebug-debugger — «как код дошёл до этой строки?», то profiler отвечает: **«почему этот запрос медленный?»**

Профилирование полезно, когда проблема не в явной ошибке, а в производительности:

- страница открывается 3-5 секунд;
- API-метод иногда «подвисает»;
- запрос к базе выполняется слишком часто;
- память растёт на больших данных;
- одна функция вызывается тысячи раз без необходимости.

#### Что измеряет profiler

| Метрика | Что показывает | Пример проблемы |
|---------|----------------|-----------------|
| **Wall time** | Реальное время выполнения | Медленный HTTP-запрос к внешнему API |
| **CPU time** | Время активной работы процессора | Тяжёлый цикл, парсинг, шифрование |
| **Memory usage** | Потребление памяти | Загрузка огромного массива целиком |
| **Call count** | Количество вызовов функции | N+1 запросы, лишние обработчики |
| **Call graph** | Кто кого вызывал | Где начинается «дорогая» цепочка |

> **На экзамене:** profiler нужен не для просмотра переменных, а для анализа **времени, памяти и частоты вызовов**.

#### Xdebug profiler

Xdebug умеет сохранять профиль выполнения в формате **Cachegrind**. Такой файл потом открывают в KCachegrind, QCachegrind, PhpStorm или других viewer-инструментах.

```ini
; php.ini (локальная разработка)
zend_extension=xdebug
xdebug.mode=debug,develop,profile
xdebug.start_with_request=trigger
xdebug.output_dir=/tmp/xdebug
xdebug.profiler_output_name=cachegrind.out.%p
```

Обычно profiling включают через trigger, а не на каждый запрос:

```bash
curl "https://example.test/catalog?XDEBUG_TRIGGER=1"
```

После запроса появляется файл вроде:

```text
cachegrind.out.12345
```

В viewer обычно смотрят:

- **Inclusive time** — время функции вместе со всеми вложенными вызовами;
- **Self time** — время только внутри самой функции;
- **Calls** — сколько раз функция была вызвана;
- **Call graph** — дерево/граф вызовов.

Пример интерпретации:

```text
ProductRepository::findByCategory()
  calls: 1
  inclusive time: 820 ms
  self time: 12 ms
```

Это значит, что сама функция почти не «думает», но внутри неё есть дорогие вложенные операции: например, SQL-запрос, ORM-гидратация или обращение к API.

#### Blackfire

**Blackfire** — специализированный profiler/APM-инструмент для PHP. Он удобнее Xdebug profiler для командной работы: умеет сравнивать профили, строить timeline, показывать bottleneck-и и хранить результаты в веб-интерфейсе.

Типичный workflow:

1. Запустить приложение в обычном окружении.
2. Снять профиль конкретного URL, CLI-команды или теста.
3. Найти самые дорогие функции по времени/памяти.
4. Оптимизировать код.
5. Снять профиль повторно и сравнить результат.

```bash
blackfire curl https://example.test/catalog
```

#### Типичный алгоритм профилирования

1. **Сформулировать симптом:** «страница каталога грузится 4 секунды».
2. **Воспроизвести сценарий:** одинаковые входные данные, одинаковый URL, прогретый/непрогретый кеш.
3. **Снять профиль:** Xdebug profiler, Blackfire, APM.
4. **Найти bottleneck:** топ функций по inclusive time, memory usage или call count.
5. **Проверить причину:** SQL, сеть, цикл, сериализация, шаблонизатор, автозагрузка.
6. **Исправить точечно:** кеш, eager loading, индекс в БД, батчинг, упрощение алгоритма.
7. **Сравнить до/после:** без повторного профиля оптимизация может быть иллюзией.

#### Частые находки profiler-а

| Симптом в профиле | Возможная причина | Что проверять |
|-------------------|-------------------|---------------|
| Много одинаковых SQL-вызовов | N+1 проблема | ORM relations, eager loading |
| Большой self time у функции | Тяжёлый алгоритм | Циклы, сортировки, регулярки |
| Большой inclusive time, маленький self time | Медленная зависимость внутри | БД, API, файловая система |
| Резкий рост memory usage | Данные копятся в массиве | Streaming, pagination, generators |
| Много вызовов шаблонизатора | Слишком сложный view | Кеширование partial-ов |

#### Важные ограничения

- Profiler замедляет приложение, особенно Xdebug, поэтому его обычно не включают постоянно.
- Профилирование на production делают осторожно: точечно, с sampling или через APM/Blackfire.
- Один профиль не всегда объективен: кеш, сеть и нагрузка могут сильно менять результат.
- Оптимизировать нужно bottleneck, а не «красивый» участок кода, который почти не влияет на время запроса.

Короткая формула:

```text
Debug показывает состояние.
Logs показывают историю.
Profiler показывает стоимость выполнения.
```

### 1.6 dd() и dump() (Laravel / Symfony VarDumper)

```php
// Laravel — остановка + красивый вывод
dd($user, $request->all());

// Только вывод, без остановки
dump($collection);

// В Blade-шаблоне
@dump($variables)
```

Symfony **VarDumper**:

```php
use Symfony\Component\VarDumper\VarDumper;

VarDumper::dump($object);
// или в dev: dd() из symfony/var-dumper
```

> `dd` = **d**ump and **d**ie — после вывода вызывается `die()`. **Только для разработки.**

### 1.7 Стек вызовов и исключения

```php
try {
    processPayment($order);
} catch (PaymentException $e) {
    // Полный trace
    error_log($e->getMessage());
    error_log($e->getTraceAsString());

    // PHP 8+: цепочка исключений
    error_log((string) $e->getPrevious()?->getMessage());
}
```

```php
// Программный trace без исключения
debug_print_backtrace(DEBUG_BACKTRACE_IGNORE_ARGS);
```

## 2. Дебаг в JavaScript: console.*

### Роль console в браузере

Объект **`console`** — API DevTools для вывода диагностической информации. Работает в браузере и в **Node.js** (с небольшими отличиями).

> **На экзамене:** `console.log` — лишь один метод; для продвинутого дебага важны **группировка, таблицы, таймеры, трассировка, уровни**.

---

### 2.1 console.log — базовый вывод

```javascript
const user = { id: 1, name: 'Anna' };

console.log('User loaded', user);
console.log('Multiple', user, document.title);

// Шаблонные строки
console.log(`Order #${orderId} total: ${total}`);
```

**Особенности:**

- Объекты выводятся **по ссылке** — при раскрытии в DevTools может быть уже изменённое состояние.
- Для «снимка» объекта: `console.log(JSON.parse(JSON.stringify(obj)))` или `structuredClone(obj)`.

---

### 2.2 Уровни и стили: warn, error, info, debug

```javascript
console.info('Информационное сообщение');
console.warn('Предупреждение: устаревший API');
console.error('Ошибка загрузки', error);
console.debug('Детали только при verbose в DevTools');
```

| Метод | Уровень в DevTools | Типичное использование |
|-------|-------------------|------------------------|
| `console.log` | Log | Общий вывод |
| `console.info` | Info | Информация |
| `console.warn` | Warning | Deprecation, странные данные |
| `console.error` | Error | Ошибки, stack (если передать Error) |
| `console.debug` | Verbose | Детальный дебаг |

```javascript
// Стилизация (только в поддерживающих браузерах)
console.log('%c API OK ', 'background: green; color: white; padding: 2px 6px;');
console.log('%c Failed ', 'background: red; color: white;');
```

---

### 2.3 console.table — табличный вывод

```javascript
const users = [
  { id: 1, name: 'Anna', role: 'admin' },
  { id: 2, name: 'Ivan', role: 'user' },
];

console.table(users);
console.table(users, ['name', 'role']); // только выбранные колонки
```

Удобно для массивов объектов и Map.

---

### 2.4 console.group / groupCollapsed / groupEnd

```javascript

console.log('Step 2: apply coupon');
console.group('Payment');

console.groupEnd();
```

`console.groupCollapsed()` — группа свёрнута по умолчанию.

### 2.5 console.time / timeLog / timeEnd — замер времени

```javascript
console.time('fetchProducts');

await fetch('/api/products');

console.timeLog('fetchProducts', 'after fetch'); // промежуточная метка
// ... обработка ...
console.timeEnd('fetchProducts'); // итог: fetchProducts: 234.567ms
```

| Метод | Действие |
|-------|----------|
| `console.time(label)` | Старт таймера |
| `console.timeLog(label, ...data)` | Промежуточное значение |
| `console.timeEnd(label)` | Стоп и вывод длительности |

### 2.6 console.trace — стек вызовов

```javascript
function a() { b(); }
function b() { c(); }
function c() {

}

// Выведет полный call stack
```

Аналог `debug_print_backtrace` в PHP.

## 3. Дебаг на проде: особенности, риски

### Почему «так лучше не делать»

**Production (прод)** — среда, где работают **реальные пользователи** и **реальные данные**. Дебаг-инструменты разработки на проде создают риски:

| Риск | Последствие |
|------|-------------|
| **Утечка данных** | `var_dump($user)` покажет email, пароли, токены |
| **Раскрытие структуры** | Stack trace → пути файлов, версии, SQL |
| **Падение производительности** | Xdebug замедляет PHP в разы |
| **Остановка запроса** | `dd()`, `die()` — 500 или пустая страница |
| **Взлом через отладчик** | Открытый Xdebug с неправильной настройкой |
| **Засорение диска** | Бесконтрольный `error_log` на миллионы строк |

> **Правило:** на проде **не дебажат вживую** как в IDE — **логируют**, **мониторят**, воспроизводят на **staging**.

---

### 3.1 Что нельзя на production

```php
// ❌ Никогда на проде
ini_set('display_errors', '1');
dd($request->all());
var_dump($_SESSION);
phpinfo();
```

```javascript
// ❌ На проде в браузере всех пользователей
console.log(userToken, creditCardData);
debugger; // остановит скрипт у того, у кого открыты DevTools
```

---

### 3.2 Что допустимо на проде

| Практика | Описание |
|----------|----------|
| **Структурированное логирование** | Monolog, PSR-3, уровни |
| **Error tracking** | Sentry, Rollbar, Bugsnag |
| **APM** | New Relic, Datadog, Blackfire |
| **Корреляционный ID** | Один id на весь HTTP-запрос |
| **display_errors = Off** | Пользователь видит generic страницу |
| **Отдельный staging** | Копия прода для воспроизведения |

```php
// ✅ Production-friendly
try {
    chargePayment($order);
} catch (PaymentException $e) {
    $logger->error('Payment failed', [
        'order_id' => $order->id,
        'exception' => $e,
        'correlation_id' => $requestId,
    ]);
    throw $e; // или безопасный ответ пользователю
}
```

---

### 3.3 Временный «горячий» дебаг на проде

Иногда без staging не обойтись. Если **вынуждены** смотреть прод:

| Правило | Зачем |
|---------|-------|
| Включать **минимально** и **кратко** | Меньше окно утечки |
| **Не** выводить в HTML/JSON ответ | Только в защищённый лог |
| Маскировать PII (email, карты) | GDPR, безопасность |
| Согласовать с DevOps | Audit trail |
| Сразу **отключить** после инцидента | Не оставлять Xdebug на неделю |

```php
// Условный дебаг только для админа + feature flag
if ($app->environment('production') && $user?->isSuperAdmin() && config('app.debug_trace')) {
    Log::debug('Trace', ['trace' => debug_backtrace(DEBUG_BACKTRACE_IGNORE_ARGS)]);
}
```

### 3.4 Xdebug на проде

**Почти всегда запрещён:** overhead 2–10×, риск подключения отладчика извне.

Если исключительно нужен:

- `xdebug.start_with_request=trigger` (не для всех запросов)
- Firewall, VPN, IP whitelist
- Отключить сразу после сессии


## 4. Логирование: уровень, сообщение, контекст

### Связь с дебагом

| Дебаг | Логирование |
|-------|-------------|
| Интерактивно, «здесь и сейчас» | Асинхронно, постфактум |
| Останавливает выполнение (`dd`) | Не должно ломать запрос |
| Часто удаляется перед коммитом | Остаётся в коде навсегда |
| Детали любые | Только **нужные** и **безопасные** данные |

**Логирование** — правильная замена «вечному дебагу» на проде: вместо `var_dump` на странице — `$logger->warning(...)` в файл/ELK/Sentry.

---

### 4.1 Стандарт PSR-3 (уровни логирования)

[PSR-3](https://www.php-fig.org/psr/psr-3/) определяет **8 уровней** (RFC 5424):

| Уровень | Когда использовать | Пример |
|---------|-------------------|--------|
| **emergency** | Система неработоспособна | БД недоступна, диск полон |
| **alert** | Нужно действовать немедленно | Платёжный шлюз лёг |
| **critical** | Критические условия | Потеря данных |
| **error** | Ошибки выполнения | Exception, failed job |
| **warning** | Не ошибка, но подозрительно | Deprecated API, retry |
| **notice** | Нормальное, но значимое | Смена конфигурации |
| **info** | Информационные события | User logged in, order created |
| **debug** | Детали для разработки | SQL, переменные (только dev/staging) |

```php
$logger->emergency('Database cluster unreachable');
$logger->error('Payment failed', ['order_id' => 42]);
$logger->warning('Rate limit approaching', ['ip' => $ip]);
$logger->info('User registered', ['user_id' => 5]);
$logger->debug('Query executed', ['sql' => $sql, 'bindings' => $bindings]);
```

**Правило уровня на проде:** `debug` и часто `info` **не пишут** или пишут в отдельный канал — иначе шум и затраты.

---

### 4.2 Сообщение (message)

**Сообщение** — человекочитаемая строка, **что произошло**. Не дублирует весь контекст.

```php
// ❌ Плохо
$logger->error(json_encode($order));

// ✅ Хорошо
$logger->error('Order payment failed', [
    'order_id' => $order->id,
    'amount' => $order->total,
    'gateway' => 'stripe',
]);
```

| Критерий | Рекомендация |
|----------|--------------|
| Статичный текст | «User login failed» |
| Динамика | В **context**, не в склейке строк |
| Язык | Один (обычно EN) для поиска в логах |
| Уникальность | Достаточно для фильтра в Kibana/Grafana |

---

### 4.3 Контекст (context)

**Контекст** — структурированные данные (массив), прилагаемые к сообщению: id сущностей, коды ошибок, метрики. Основа **structured logging**.

```php
$logger->info('Order shipped', [
    'order_id'    => $order->id,
    'user_id'     => $order->user_id,
    'carrier'     => 'cdek',
    'tracking'    => $trackNumber,
    'duration_ms' => $duration,
    'request_id'  => $requestId,
]);
```

**JSON в логе** (удобно для парсинга):

```json
{
  "level": "info",
  "message": "Order shipped",
  "context": {
    "order_id": 1001,
    "user_id": 42
  },
  "datetime": "2026-05-27T14:30:00+00:00",
  "channel": "app"
}
```

### Что не класть в контекст на проде

| Нельзя | Можно (маскированное) |
|--------|------------------------|
| Пароли, plain tokens | `user_id`, `order_id` |
| Номера карт CVV | `card_last4` |
| Полный JWT | `jti`, hash |
| Сессии целиком | `session_id` (осторожно) |

```php
$logger->info('Login attempt', [
    'email' => maskEmail($user->email), // a***@mail.ru
    'ip' => $request->ip(),
]);
```

---

### 4.4 Monolog в PHP (практика)

**Monolog** — де-факто стандарт, реализует PSR-3.

```php
use Monolog\Logger;
use Monolog\Handler\StreamHandler;
use Monolog\Handler\RotatingFileHandler;

$log = new Logger('app');
$log->pushHandler(new RotatingFileHandler('/var/log/app/app.log', 14, Logger::DEBUG));
$log->pushHandler(new StreamHandler('php://stderr', Logger::ERROR));

$log->info('Application started');
$log->error('Exception caught', ['exception' => $e]);
```

| Handler | Назначение |
|---------|------------|
| `StreamHandler` | Файл или поток |
| `RotatingFileHandler` | Ротация по дням |
| `SyslogHandler` | syslog |
| `ErrorLogHandler` | PHP error_log |
| Интеграции | Slack, Sentry, Elasticsearch |

**Laravel** использует Monolog под капотом:

```php
Log::channel('daily')->error('Failed job', ['job' => $jobName]);
```

---

### 4.5 Централизованное логирование: ELK

**ELK** — популярная схема централизованного логирования:

```text
PHP (Monolog)
  -> файл / сокет / HTTP
  -> Filebeat или Logstash
  -> Elasticsearch
  -> Kibana
```

Главная идея: приложение не хранит и не анализирует логи само. Оно только **создаёт события**, а отдельная инфраструктура собирает, обрабатывает, хранит и показывает их.

| Компонент | Роль |
|-----------|------|
| **Monolog** | Формирует лог-сообщение в PHP |
| **Filebeat** | Лёгкий агент: читает файлы логов и отправляет дальше |
| **Logstash** | Промежуточный обработчик: парсит, фильтрует, обогащает |
| **Elasticsearch** | Хранит и индексирует логи для быстрого поиска |
| **Kibana** | Веб-интерфейс для поиска, фильтров, графиков и дашбордов |

Чаще всего PHP пишет лог в файл:

```php
$logger->error('Profile loading failed', [
    'user_id' => 42,
    'url' => '/profile',
    'exception' => $e->getMessage(),
]);
```

В файле это может быть JSON-строка:

```json
{
  "level": "ERROR",
  "message": "Profile loading failed",
  "user_id": 42,
  "url": "/profile",
  "datetime": "2026-06-17 23:20:00"
}
```

Дальше путь такой:

```text
1. Monolog записал ошибку в app.log
2. Filebeat прочитал новую строку из файла
3. Logstash разобрал лог и привёл его к удобной структуре
4. Elasticsearch сохранил и проиндексировал запись
5. Kibana показала ошибку в интерфейсе
```

Пример поиска в Kibana:

```text
level: ERROR AND user_id: 42
```

Так разработчик может быстро найти ошибку по пользователю, URL, времени, `request_id`, уровню лога или тексту исключения.

> **На экзамене:** Monolog — это библиотека в приложении, Filebeat/Logstash — доставка и обработка, Elasticsearch — хранение и поиск, Kibana — просмотр.

---

### 4.6 Каналы, процессоры, форматтеры

```
Logger
  ├── Handler (куда пишем: file, sentry)
  ├── Formatter (как: JSON, Line)
  └── Processor (добавить request_id, memory, user)
```

```php
// Процессор добавляет request_id ко всем записям
$log->pushProcessor(function ($record) {
    $record['extra']['request_id'] = $_SERVER['HTTP_X_REQUEST_ID'] ?? uniqid();
    return $record;
});
```

---

### 4.7 Корреляция запросов

Один **request_id** / **correlation_id** во всех логах одного HTTP-запроса:

```php
// Middleware
$requestId = $request->header('X-Request-ID') ?? Str::uuid()->toString();
Log::withContext(['request_id' => $requestId]);

// Все последующие Log::info автоматически с request_id
```

Упрощает поиск: «покажи все логи по заказу #1001 за этот запрос».

### 4.8 Связь логирования с дебагом (итоговая схема)

```
┌─────────────────────────────────────────────────────────────┐
│  Локальная разработка                                        │
│  • Xdebug, dd(), console.table                               │
│  • Уровень debug в логах                                     │
└───────────────────────────┬─────────────────────────────────┘
                            │ баг найден
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  Исправление + постоянные логи info/warning/error            │
│  • Убрать dd/var_dump                                        │
│  • Оставить $logger->error с контекстом                      │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  Production                                                  │
│  • Только логи + Sentry/APM                                  │
│  • display_errors Off, debug level Off                       │
│  • Алерты на critical/error                                  │
└─────────────────────────────────────────────────────────────┘
```

| Ситуация | Дебаг | Логирование |
|----------|-------|-------------|
| «Почему падает у меня локально?» | Xdebug, dd | debug в файл |
| «Почему упало у пользователя вчера?» | Staging + replay | error + context + request_id |
| «Сайт медленный» | Profiler | info/warning + APM |

### 4.9 Пример: единый подход в приложении

```php
<?php
declare(strict_types=1);

final class OrderService
{
    public function __construct(
        private LoggerInterface $logger,
    ) {}

    public function placeOrder(array $data): Order
    {
        $this->logger->info('Placing order', ['user_id' => $data['user_id']]);

        try {
            $order = $this->repository->create($data);
            $this->logger->info('Order created', ['order_id' => $order->id]);
            return $order;
        } catch (Throwable $e) {
            $this->logger->error('Order creation failed', [
                'user_id' => $data['user_id'],
                'exception' => $e,
            ]);
            throw $e;
        }
    }
}
```

```javascript
// Frontend: ошибки на сервер (не console.log на проде)
window.addEventListener('error', (event) => {
  fetch('/api/client-log', {
    method: 'POST',
    body: JSON.stringify({
      message: event.message,
      stack: event.error?.stack,
      url: location.href,
    }),
  });
});
```

## Шпаргалка перед экзаменом

| Вопрос | Ответ в одну строку |
|--------|---------------------|
| Дебаг в PHP | var_dump, Xdebug, dd — для **разработки** |
| console.* | Не только log: table, group, time, trace, assert |
| Дебаг на проде | **Опасно**: утечки, производительность; использовать **логи + мониторинг** |
| Уровень лога | emergency → debug (PSR-3) |
| Сообщение | Что случилось (текст) |
| Контекст | Структурированные данные (массив/JSON) |
| Связь | Дебаг находит баг → логирование **остаётся** для прода |

### Типичные ошибки

| Ошибка | Правильно |
|--------|-----------|
| `display_errors=On` на проде | Off + log_errors |
| Пароль в `console.log` | Никогда; маскировать в логах |
| `dd()` в API на проде | Logger + generic response |
| Логировать всё на уровне debug | Фильтр по среде |
| Сообщение = `print_r($obj)` | Message + context |
