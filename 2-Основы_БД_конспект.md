## Кратко: что такое реляционная БД

**База данных (БД)** — организованное хранилище данных.  
**Реляционная БД** хранит данные в **таблицах** (строки и столбцы). Связи между сущностями задаются через **ключи** (первичный, внешний).

| Термин | Значение |
|--------|----------|
| **Таблица (table)** | Набор записей одного типа |
| **Строка (row / record)** | Одна запись (один объект) |
| **Столбец (column / field)** | Одно поле (атрибут) |
| **Первичный ключ (PK)** | Уникальный идентификатор строки |
| **Внешний ключ (FK)** | Ссылка на PK другой таблицы |
| **SQL** | Язык запросов к реляционным БД |

## 1. Создание таблиц

### Команда CREATE TABLE

Таблица создаётся командой **`CREATE TABLE`**: задаются имена столбцов, типы данных и ограничения.

```sql
CREATE TABLE users (
    id          INT UNSIGNED NOT NULL AUTO_INCREMENT,
    email       VARCHAR(255) NOT NULL,
    name        VARCHAR(100) NOT NULL,
    age         TINYINT UNSIGNED NULL,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_users_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### Основные типы данных (MySQL)

| Тип | Назначение | Пример значения |
|-----|------------|-----------------|
| `INT`, `BIGINT` | Целые числа | `42`, `1000` |
| `DECIMAL(10,2)` | Деньги, точные дроби | `1999.99` |
| `FLOAT`, `DOUBLE` | Приближённые дроби | `3.14` |
| `VARCHAR(n)` | Строка переменной длины | `'Иван'` |
| `TEXT` | Длинный текст | описание товара |
| `DATE` | Дата | `'2026-05-27'` |
| `DATETIME`, `TIMESTAMP` | Дата и время | `'2026-05-27 14:30:00'` |
| `BOOLEAN` (TINYINT(1)) | Логическое | `TRUE` / `FALSE` |
| `ENUM('a','b')` | Одно из фиксированных значений | `'active'` |

### Частые ограничения (constraints)

| Ограничение | Смысл |
|-------------|-------|
| `PRIMARY KEY` | Уникальный идентификатор строки |
| `NOT NULL` | Поле обязательно к заполнению |
| `UNIQUE` | Все значения в столбце уникальны |
| `DEFAULT` | Значение по умолчанию |
| `AUTO_INCREMENT` | Автонумерация (обычно для `id`) |
| `FOREIGN KEY` | Ссылка на другую таблицу |

### Создание таблицы со связью (внешний ключ)

```sql
CREATE TABLE orders (
    id          INT UNSIGNED NOT NULL AUTO_INCREMENT,
    user_id     INT UNSIGNED NOT NULL,
    total       DECIMAL(10, 2) NOT NULL DEFAULT 0,
    status      ENUM('new', 'paid', 'shipped') NOT NULL DEFAULT 'new',
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_orders_user_id (user_id),
    CONSTRAINT fk_orders_user
        FOREIGN KEY (user_id) REFERENCES users (id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### Изменение структуры существующей таблицы

```sql
-- Добавить столбец
ALTER TABLE users ADD COLUMN phone VARCHAR(20) NULL AFTER email;

-- Изменить тип
ALTER TABLE users MODIFY COLUMN name VARCHAR(150) NOT NULL;

-- Удалить столбец
ALTER TABLE users DROP COLUMN phone;
```

### Вставка данных при создании (опционально)

После `CREATE TABLE` данные добавляют через `INSERT` (см. примеры в разделе 2).

```sql
INSERT INTO users (email, name, age) VALUES
    ('anna@mail.ru', 'Анна', 25),
    ('ivan@mail.ru', 'Иван', 30);
```

## 2. Выборка данных

### Базовый SELECT

**`SELECT`** — чтение данных из таблицы. Это самая частая SQL-команда.

```sql
-- Все столбцы
SELECT * FROM users;

-- Конкретные столбцы
SELECT id, name, email FROM users;

-- Псевдонимы столбцов (AS)
SELECT name AS user_name, email AS user_email FROM users;
```

### Фильтрация: WHERE

```sql
SELECT * FROM users WHERE age >= 18;

SELECT id, name FROM users
WHERE email LIKE '%@mail.ru'
  AND age BETWEEN 20 AND 40;

SELECT * FROM orders WHERE status IN ('new', 'paid');

SELECT * FROM users WHERE phone IS NULL;
```

| Оператор | Пример | Смысл |
|----------|--------|-------|
| `=`, `<>`, `!=` | `status = 'new'` | Равно / не равно |
| `>`, `<`, `>=`, `<=` | `age >= 18` | Сравнение |
| `AND`, `OR`, `NOT` | `a AND b` | Логические связки |
| `IN (...)` | `status IN ('new','paid')` | Входит в список |
| `BETWEEN a AND b` | `age BETWEEN 18 AND 65` | Диапазон |
| `LIKE` | `name LIKE 'Ив%'` | Шаблон (`%` — любая строка) |
| `IS NULL` | `phone IS NULL` | Проверка на NULL |

### Сортировка и ограничение

```sql
-- Сортировка
SELECT * FROM users ORDER BY name ASC;
SELECT * FROM orders ORDER BY created_at DESC, id DESC;

-- Пагинация (LIMIT offset, count)
SELECT * FROM users ORDER BY id LIMIT 10;        -- первые 10
SELECT * FROM users ORDER BY id LIMIT 10, 10;   -- пропустить 10, взять 10
-- В MySQL 8+: LIMIT 10 OFFSET 10;
```

### Агрегатные функции и GROUP BY

```sql
-- Количество пользователей
SELECT COUNT(*) AS total_users FROM users;

-- Сумма заказов по пользователю
SELECT user_id, COUNT(*) AS order_count, SUM(total) AS total_spent
FROM orders
GROUP BY user_id
HAVING total_spent > 1000;
```

| Функция | Назначение |
|---------|------------|
| `COUNT()` | Количество строк |
| `SUM()` | Сумма |
| `AVG()` | Среднее |
| `MIN()`, `MAX()` | Минимум / максимум |

> **`WHERE`** фильтрует строки **до** группировки.  
> **`HAVING`** фильтрует **после** `GROUP BY`.

### Уникальные значения

```sql
SELECT DISTINCT status FROM orders;
```

### Подзапросы (кратко)

```sql
SELECT name FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total > 5000);
```

## 3. Обновление данных

### Команда UPDATE

**`UPDATE`** изменяет существующие строки. **Всегда используйте `WHERE`**, иначе обновятся все записи таблицы.

```sql
UPDATE users
SET name = 'Анна Петрова', age = 26
WHERE id = 1;
```

### Обновление нескольких полей

```sql
UPDATE orders
SET status = 'paid',
    total = total - 100
WHERE id = 42 AND status = 'new';
```

### Обновление по условию из другой таблицы (JOIN)

```sql
UPDATE orders o
INNER JOIN users u ON u.id = o.user_id
SET o.status = 'cancelled'
WHERE u.email = 'blocked@mail.ru';
```

### UPDATE с выражениями

```sql
-- Увеличить цену всех товаров категории на 10%
UPDATE products
SET price = price * 1.10
WHERE category_id = 5;
```

### Безопасная практика

1. Сначала проверьте выборкой, **какие строки затронуты**:

```sql
SELECT * FROM users WHERE id = 1;
```

2. Затем выполните `UPDATE` с тем же `WHERE`.
3. В транзакциях: `START TRANSACTION` → `UPDATE` → `COMMIT` или `ROLLBACK`.

```sql
START TRANSACTION;

UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;

COMMIT;
```

## 4. Удаление данных

### Команда DELETE

**`DELETE`** удаляет **строки** из таблицы. Структура таблицы сохраняется.

```sql
DELETE FROM users WHERE id = 10;
```

### Удаление нескольких записей

```sql
DELETE FROM orders
WHERE status = 'cancelled'
  AND created_at < '2025-01-01';
```

### DELETE с JOIN (MySQL)

```sql
DELETE o FROM orders o
INNER JOIN users u ON u.id = o.user_id
WHERE u.email = 'spam@test.ru';
```

### DELETE vs TRUNCATE

| Команда | Что удаляет | WHERE | Откат (в транзакции) |
|---------|-------------|-------|----------------------|
| `DELETE` | Строки по условию | Да | Да (InnoDB) |
| `TRUNCATE` | **Все** строки таблицы | Нет | Зависит от СУБД |

```sql
-- Осторожно: удалит ВСЕ строки!
DELETE FROM logs;

-- Быстро очистить таблицу целиком (сброс AUTO_INCREMENT)
TRUNCATE TABLE logs;
```

### Каскадное удаление (при FK)

Если при создании FK указано `ON DELETE CASCADE`, удаление родителя удалит связанные дочерние строки:

```sql
-- При удалении user удалятся его orders
CONSTRAINT fk_orders_user
    FOREIGN KEY (user_id) REFERENCES users (id)
    ON DELETE CASCADE
```

```sql
DELETE FROM users WHERE id = 5;
-- автоматически удалятся orders с user_id = 5
```

## 5. Удаление таблиц

### Команда DROP TABLE

**`DROP TABLE`** удаляет таблицу **целиком**: и данные, и структуру. Операция необратима (без резервной копии).

```sql
DROP TABLE temp_import;
```

### Удаление нескольких таблиц

```sql
DROP TABLE IF EXISTS old_logs, old_sessions, temp_data;
```

### DROP vs DELETE vs TRUNCATE

| Команда | Структура таблицы | Данные | Скорость на больших таблицах |
|---------|-------------------|--------|------------------------------|
| `DELETE` | Сохраняется | Удаляются строки (выборочно) | Медленнее |
| `TRUNCATE` | Сохраняется | Удаляются **все** строки | Быстро |
| `DROP TABLE` | **Удаляется** | Удаляются | — |

### Удаление с проверкой зависимостей

Если на таблицу ссылаются внешние ключи, `DROP` может завершиться ошибкой. Варианты:

```sql
-- Сначала удалить зависимые таблицы или снять FK
DROP TABLE orders;
DROP TABLE users;

-- Либо (осторожно!) временно отключить проверки FK:
SET FOREIGN_KEY_CHECKS = 0;
DROP TABLE users;
SET FOREIGN_KEY_CHECKS = 1;
```

### Переименование вместо удаления (альтернатива)

```sql
RENAME TABLE users TO users_archive;
-- MySQL: RENAME TABLE old TO new;
```

## 6. Отношения между таблицами

### Зачем нужны связи

Данные разбивают на несколько таблиц, чтобы **избежать дублирования** и обеспечить **целостность**. Например, имя пользователя хранится один раз в `users`, а заказы ссылаются на него через `user_id`.

### Типы связей

```
Один ко многим (1:N)          Многие ко многим (N:M)
┌─────────┐                   ┌─────────┐     ┌──────────────┐     ┌─────────┐
│  users  │                   │ students│─────│ enrollments  │─────│ courses │
│   id    │◄──┐               └─────────┘     └──────────────┘     └─────────┘
└─────────┘   │               (связующая / junction table)
              │
         ┌────┴────┐
         │ orders  │
         │ user_id │
         └─────────┘

Один к одному (1:1)
┌─────────┐     ┌──────────────┐
│  users  │────►│ user_profiles│
│   id    │     │   user_id    │
└─────────┘     └──────────────┘
```

| Тип | Описание | Пример |
|-----|----------|--------|
| **1:N** | Одна запись A — много записей B | Один пользователь — много заказов |
| **N:M** | Много A связаны с многими B | Студенты и курсы (через таблицу связи) |
| **1:1** | Строго одна пара | Пользователь и его профиль |

### Первичный и внешний ключ

```sql
-- Родительская таблица
CREATE TABLE categories (
    id   INT UNSIGNED NOT NULL AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    PRIMARY KEY (id)
);

-- Дочерняя таблица (связь 1:N)
CREATE TABLE products (
    id          INT UNSIGNED NOT NULL AUTO_INCREMENT,
    category_id INT UNSIGNED NOT NULL,
    title       VARCHAR(200) NOT NULL,
    price       DECIMAL(10, 2) NOT NULL,
    PRIMARY KEY (id),
    CONSTRAINT fk_products_category
        FOREIGN KEY (category_id) REFERENCES categories (id)
        ON DELETE RESTRICT   -- нельзя удалить категорию, если есть товары
        ON UPDATE CASCADE    -- при смене id категории обновится category_id
);
```

### Поведение ON DELETE / ON UPDATE

| Действие | Поведение |
|----------|-----------|
| `RESTRICT` / `NO ACTION` | Запретить операцию, если есть связанные строки |
| `CASCADE` | Каскадно удалить / обновить дочерние строки |
| `SET NULL` | В дочерней записи FK станет `NULL` |
| `SET DEFAULT` | Подставить значение по умолчанию |

### Связь «многие ко многим» (N:M)

Нужна **промежуточная таблица** с двумя внешними ключами:

```sql
CREATE TABLE students (
    id   INT UNSIGNED NOT NULL AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE courses (
    id   INT UNSIGNED NOT NULL AUTO_INCREMENT,
    title VARCHAR(150) NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE enrollments (
    student_id INT UNSIGNED NOT NULL,
    course_id  INT UNSIGNED NOT NULL,
    enrolled_at DATE NOT NULL,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students (id) ON DELETE CASCADE,
    FOREIGN KEY (course_id)  REFERENCES courses (id) ON DELETE CASCADE
);
```

### Нормализация (кратко для экзамена)

| Нормальная форма | Идея |
|------------------|------|
| **1NF** | Атомарные значения в ячейках, нет повторяющихся групп |
| **2NF** | Нет зависимости от части составного ключа |
| **3NF** | Нет транзитивных зависимостей (поле зависит только от PK) |

## 7. Выборка из нескольких связанных таблиц

### Зачем JOIN

Когда данные разнесены по таблицам, **`JOIN`** объединяет строки по условию связи (обычно `таблица_a.pk = таблица_b.fk`).

### INNER JOIN

Возвращает только строки, для которых **есть совпадение в обеих таблицах**.

```sql
SELECT
    u.id,
    u.name,
    o.id AS order_id,
    o.total,
    o.status
FROM users u
INNER JOIN orders o ON o.user_id = u.id
WHERE o.status = 'paid'
ORDER BY o.created_at DESC;
```

### LEFT JOIN (LEFT OUTER JOIN)

Все строки из **левой** таблицы + совпадения справа. Если совпадения нет — столбцы правой таблицы будут `NULL`.

```sql
-- Все пользователи, даже без заказов
SELECT
    u.id,
    u.name,
    COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;
```

### RIGHT JOIN

Зеркало `LEFT JOIN`: все строки из **правой** таблицы. На практике чаще переписывают через `LEFT JOIN`, меняя порядок таблиц.

```sql
SELECT u.name, o.id
FROM orders o
RIGHT JOIN users u ON u.id = o.user_id;
-- Эквивалентно: FROM users u LEFT JOIN orders o ...
```

### Сравнение типов JOIN

| Тип | Что попадает в результат |
|-----|--------------------------|
| `INNER JOIN` | Только совпавшие пары |
| `LEFT JOIN` | Все слева + совпадения справа |
| `RIGHT JOIN` | Все справа + совпадения слева |
| `FULL OUTER JOIN` | Все с обеих сторон (в MySQL эмулируют UNION) |

### JOIN трёх и более таблиц

```sql
SELECT
    u.name AS customer,
    o.id AS order_id,
    p.title AS product,
    oi.quantity,
    oi.price
FROM orders o
INNER JOIN users u       ON u.id = o.user_id
INNER JOIN order_items oi ON oi.order_id = o.id
INNER JOIN products p    ON p.id = oi.product_id
WHERE o.created_at >= '2026-01-01'
ORDER BY o.id, p.title;
```

### Псевдонимы таблиц

Короткие алиасы (`u`, `o`, `p`) делают запрос читаемым и обязательны при одноимённых столбцах:

```sql
SELECT u.name, o.total
FROM users u
JOIN orders o ON o.user_id = u.id;
```

### Неявный JOIN (устаревший стиль)

```sql
-- Старый синтаксис через WHERE (знайте, но на экзамене лучше явный JOIN)
SELECT u.name, o.total
FROM users u, orders o
WHERE o.user_id = u.id;
```

### Подзапрос vs JOIN

Один и тот же результат часто можно получить двумя способами:

```sql
-- Подзапрос
SELECT name FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total > 10000);

-- JOIN
SELECT DISTINCT u.name
FROM users u
INNER JOIN orders o ON o.user_id = u.id
WHERE o.total > 10000;
```

`JOIN` обычно предпочтительнее для читаемости и оптимизации.

### Практический пример: интернет-магазин

```sql
-- Схема
CREATE TABLE users (id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100));
CREATE TABLE products (id INT PRIMARY KEY AUTO_INCREMENT, title VARCHAR(200), price DECIMAL(10,2));
CREATE TABLE orders (id INT PRIMARY KEY AUTO_INCREMENT, user_id INT, FOREIGN KEY (user_id) REFERENCES users(id));
CREATE TABLE order_items (
    order_id INT, product_id INT, qty INT,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

-- Отчёт: кто что заказал
SELECT
    u.name,
    o.id AS order_id,
    p.title,
    oi.qty,
    p.price * oi.qty AS line_total
FROM users u
JOIN orders o       ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p     ON p.id = oi.product_id;
```

### Диаграмма JOIN (наглядно)

```
users (LEFT)          orders (RIGHT)

INNER JOIN  →  только пересечение (есть и user, и order)
LEFT JOIN   →  все users + их orders (или NULL)
```

## Шпаргалка: основные команды SQL

| Задача | Команда |
|--------|---------|
| Создать таблицу | `CREATE TABLE ...` |
| Добавить данные | `INSERT INTO ... VALUES ...` |
| Прочитать данные | `SELECT ... FROM ... WHERE ...` |
| Изменить данные | `UPDATE ... SET ... WHERE ...` |
| Удалить строки | `DELETE FROM ... WHERE ...` |
| Очистить таблицу | `TRUNCATE TABLE ...` |
| Удалить таблицу | `DROP TABLE ...` |
| Связать таблицы | `FOREIGN KEY ... REFERENCES ...` |
| Объединить в запросе | `INNER JOIN`, `LEFT JOIN` |

### Порядок выполнения SELECT (логический)

```
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

Понимание этого порядка помогает на экзамене объяснить, почему нельзя использовать алиас из `SELECT` в `WHERE` (в стандартном SQL), но можно в `ORDER BY`.

## Типичные ошибки на экзамене

| Ошибка | Правильно |
|--------|-----------|
| `UPDATE users SET age = 20` без `WHERE` | Всегда указывать условие |
| `DELETE users` | `DELETE FROM users WHERE ...` |
| Путать `DELETE` и `DROP TABLE` | `DELETE` — строки, `DROP` — вся таблица |
| `WHERE age = NULL` | `WHERE age IS NULL` |
| Забыть индекс на FK | `KEY idx_orders_user_id (user_id)` |
| N:M без связующей таблицы | Нужна таблица с двумя FK |
