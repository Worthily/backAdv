## Введение: зачем нужна система событий

В монолитном коде модули жёстко вызывают друг друга:

```
Заказ оформлен → отправить email → записать в лог → обновить склад → ...
```

Каждая новая функция требует **правки ядра**. Система событий разрывает эту связь: ядро **сообщает**, что что-то произошло, а сторонние модули **сами решают**, реагировать или нет.

```
┌──────────────┐    событие "OrderCreated"    ┌─────────────────┐
│  Ядро /      │ ───────────────────────────► │  Подписчики     │
│  Контроллер  │                                │  (слушатели)    │
└──────────────┘                                └─────────────────┘
                                                        │
                        ┌───────────────────────────────┼───────────────────────────────┐
                        ▼                               ▼                               ▼
                  EmailListener                  StockListener                    LogListener
```

Это реализация паттерна **Observer (Наблюдатель)** и принципа **слабой связанности (loose coupling)**.

## 1. События и хуки — что это

### Событие (Event)

**Событие** — факт, что в приложении **что-то произошло**: пользователь зарегистрировался, заказ создан, статья опубликована. Событие обычно представлено **объектом** или **именованным сигналом** с данными (payload).

```php
// Простое событие как объект (Laravel-стиль)
class OrderCreated
{
    public function __construct(
        public int $orderId,
        public int $userId,
        public float $total
    ) {}
}
```

**Диспетчер событий (Event Dispatcher)** — компонент, который **рассылает** событие всем зарегистрированным обработчикам.

### Хук (Hook)

**Хук** — точка в коде ядра, куда можно **«подцепиться»** и выполнить свой код **без изменения исходников**. Термин чаще встречается в **CMS** (WordPress, Drupal).

| Термин CMS | Аналог в фреймворках |
|------------|----------------------|
| **Action** (действие) | Событие без возврата значения — «сделай что-то после X» |
| **Filter** (фильтр) | Событие с изменением данных — «преобразуй значение и верни» |
| **Hook** | Общее название точки подключения |
| **Listener / Subscriber** | Обработчик события |

> На экзамене: **событие** — что произошло; **хук** — место/механизм, куда вешается обработчик. В разговорной речи термины часто смешивают.

### Типы взаимодействия

| Тип | Поведение | Пример |
|-----|-----------|--------|
| **Синхронный** | Обработчики выполняются сразу, в том же запросе | WordPress `do_action`, Laravel `event()` по умолчанию |
| **Асинхронный** | Обработка отложена (очередь, cron) | Laravel `ShouldQueue`, Symfony Messenger |
| **Filter / Middleware** | Данные проходят через цепочку, каждый может изменить | WordPress `apply_filters` |

### Минимальная реализация «на пальцах»

```php
/**
 * Упрощённый диспетчер событий — для понимания идеи, не production-код.
 */
class EventDispatcher
{
    /** @var array<string, callable[]> */
    private array $listeners = [];

    public function on(string $eventName, callable $listener): void
    {
        $this->listeners[$eventName][] = $listener;
    }

    public function dispatch(string $eventName, mixed $payload = null): void
    {
        foreach ($this->listeners[$eventName] ?? [] as $listener) {
            $listener($payload);
        }
    }
}

// Использование
$dispatcher = new EventDispatcher();

$dispatcher->on('user.registered', function (array $user) {
    mail($user['email'], 'Добро пожаловать', 'Вы зарегистрированы');
});

$dispatcher->on('user.registered', function (array $user) {
    error_log('Новый пользователь: ' . $user['email']);
});

$dispatcher->dispatch('user.registered', [
    'id' => 1,
    'email' => 'anna@mail.ru',
]);
```

### Filter (изменение данных по цепочке)

```php
class FilterChain
{
    private array $filters = [];

    public function add(string $name, callable $filter): void
    {
        $this->filters[$name][] = $filter;
    }

    public function apply(string $name, mixed $value, mixed ...$args): mixed
    {
        foreach ($this->filters[$name] ?? [] as $filter) {
            $value = $filter($value, ...$args);
        }
        return $value;
    }
}

$filters = new FilterChain();

$filters->add('post.title', fn(string $title) => strip_tags($title));
$filters->add('post.title', fn(string $title) => mb_strtoupper($title));

$title = $filters->apply('post.title', '<b>Hello</b>');
// HELLO
```

### Связь с паттернами

| Паттерн | Роль |
|---------|------|
| **Observer** | Подписчики реагируют на изменение субъекта |
| **Mediator** | Диспетчер координирует, кто кого вызывает |
| **Pub/Sub** | Издатель не знает подписчиков, только канал/топик |

Ниже — три распространённых экосистемы. На экзамене достаточно **уверенно описать одну** (часто WordPress или Laravel) и знать, что аналоги есть везде.

---

### 2.1 WordPress — классическая система хуков

WordPress построен на **Actions** и **Filters**. Плагины и темы почти не трогают ядро — только цепляются к хукам.

#### Actions (события)

```php
// Регистрация обработчика (подписка)
add_action('init', 'my_plugin_init');
add_action('wp_enqueue_scripts', 'my_plugin_load_assets');

function my_plugin_init(): void
{
    // код при инициализации WordPress
}

function my_plugin_load_assets(): void
{
    wp_enqueue_style('my-style', plugins_url('style.css', __FILE__));
}

// В ядре (упрощённо) в нужный момент вызывается:
do_action('init');
```

| Функция | Назначение |
|---------|------------|
| `add_action($hook, $callback, $priority, $args)` | Подписаться на действие |
| `do_action($hook, ...$args)` | Вызвать все обработчики |
| `remove_action($hook, $callback, $priority)` | Отписаться |

**Приоритет** — порядок выполнения (меньше число → раньше). По умолчанию `10`.

```php
add_action('save_post', 'notify_admin', 5);   // выполнится раньше
add_action('save_post', 'update_cache', 20); // выполнится позже
```

#### Filters (преобразование данных)

```php
// Подписаться на фильтр
add_filter('the_title', 'make_title_uppercase');

function make_title_uppercase(string $title): string
{
    return mb_strtoupper($title);
}

// В ядре при выводе заголовка:
$title = apply_filters('the_title', $post->post_title);
echo $title;
```

| Функция | Назначение |
|---------|------------|
| `add_filter($hook, $callback, $priority, $args)` | Добавить фильтр |
| `apply_filters($hook, $value, ...$args)` | Прогнать значение через цепочку |
| `remove_filter(...)` | Убрать фильтр |

#### Популярные хуки (для экзамена)

| Хук | Когда срабатывает |
|-----|-------------------|
| `init` | WordPress загружен, можно регистрировать типы записей |
| `wp_enqueue_scripts` | Подключение CSS/JS на фронте |
| `save_post` | После сохранения записи |
| `the_content` | Перед выводом содержимого поста |
| `plugins_loaded` | Все плагины загружены |

#### Схема жизненного цикла (упрощённо)

```
plugins_loaded → init → wp_loaded → template_redirect → wp_enqueue_scripts → ...
                                                      ↓
                                            the_content, the_title (filters)
```

### 2.2 Laravel — Events, Listeners, Observers

Laravel использует объектно-ориентированную модель: **класс события** + **слушатели (listeners)**.

#### Событие и слушатель

```php
// app/Events/OrderCreated.php
namespace App\Events;

use Illuminate\Foundation\Events\Dispatchable;
use Illuminate\Queue\SerializesModels;

class OrderCreated
{
    use Dispatchable, SerializesModels;

    public function __construct(public \App\Models\Order $order) {}
}
```

```php
// app/Listeners/SendOrderConfirmation.php
namespace App\Listeners;

use App\Events\OrderCreated;
use Illuminate\Support\Facades\Mail;

class SendOrderConfirmation
{
    public function handle(OrderCreated $event): void
    {
        Mail::to($event->order->user->email)
            ->send(new \App\Mail\OrderPlaced($event->order));
    }
}
```

#### Регистрация в EventServiceProvider

```php
// app/Providers/EventServiceProvider.php
protected $listen = [
    \App\Events\OrderCreated::class => [
        \App\Listeners\SendOrderConfirmation::class,
        \App\Listeners\UpdateStock::class,
    ],
];
```

#### Вызов события

```php
// В контроллере или сервисе после создания заказа
use App\Events\OrderCreated;

$order = Order::create($data);
event(new OrderCreated($order));

// или
OrderCreated::dispatch($order);
```

#### Асинхронная обработка (очередь)

```php
class SendOrderConfirmation implements \Illuminate\Contracts\Queue\ShouldQueue
{
    public function handle(OrderCreated $event): void
    {
        // выполнится воркером очереди, не в HTTP-запросе
    }
}
```

#### Observers (наблюдатели модели Eloquent)

Реагируют на **жизненный цикл модели** без явного `event()`:

```php
// app/Observers/UserObserver.php
class UserObserver
{
    public function created(User $user): void
    {
        // автоматически при User::create()
    }

    public function deleting(User $user): void
    {
        // перед удалением
    }
}

// AppServiceProvider::boot()
User::observe(UserObserver::class);
```

| Механизм | Когда использовать |
|----------|-------------------|
| **Events + Listeners** | Бизнес-события: «заказ создан», «оплата прошла» |
| **Observers** | CRUD модели: created, updated, deleted |
| **Middleware** | HTTP-запрос/ответ, не доменные события |

### Источники Laravel

- [Laravel Docs: Events](https://laravel.com/docs/events)
- [Laravel Docs: Observers](https://laravel.com/docs/eloquent#observers)
- [Laravel Docs: Queues](https://laravel.com/docs/queues)

### 2.3 Symfony — EventDispatcher

Компонент **EventDispatcher** — эталон для многих PHP-проектов.

```php
use Symfony\Component\EventDispatcher\EventDispatcher;
use Symfony\Contracts\EventDispatcher\Event;

// Событие
class OrderPlacedEvent extends Event
{
    public const NAME = 'order.placed';

    public function __construct(private Order $order) {}

    public function getOrder(): Order
    {
        return $this->order;
    }
}

// Слушатель
class OrderEmailListener
{
    public function onOrderPlaced(OrderPlacedEvent $event): void
    {
        $order = $event->getOrder();
        // отправка письма
    }
}

// Регистрация
$dispatcher = new EventDispatcher();
$dispatcher->addListener(
    OrderPlacedEvent::NAME,
    [new OrderEmailListener(), 'onOrderPlaced']
);

// Диспетчеризация
$dispatcher->dispatch(new OrderPlacedEvent($order), OrderPlacedEvent::NAME);
```

В полном Symfony подписчики регистрируются через **`EventSubscriberInterface`** с методом `getSubscribedEvents()`.

```php
use Symfony\Component\EventDispatcher\EventSubscriberInterface;

class OrderSubscriber implements EventSubscriberInterface
{
    public static function getSubscribedEvents(): array
    {
        return [
            OrderPlacedEvent::NAME => 'onOrderPlaced',
            PaymentReceivedEvent::NAME => ['onPayment', 10],
        ];
    }

    public function onOrderPlaced(OrderPlacedEvent $event): void { /* ... */ }
}
```

| | WordPress | Laravel | Symfony |
|---|-----------|---------|---------|
| **Терминология** | Actions, Filters, Hooks | Events, Listeners, Observers | Events, Listeners, Subscribers |

### 2.5 Типичные сценарии использования

| Сценарий | Пример события / хука |
|----------|------------------------|
| Уведомления | После регистрации → email, SMS |
| Аналитика | `order.paid` → отправка в Metrika/GA |
| Кэш | `save_post` → сброс кэша страницы |
| Аудит / лог | Любое действие админа → запись в log |
| Расширение без форка | Плагин меняет вывод через `the_content` |
| Микросервисы (позже) | Событие в шину (RabbitMQ, Kafka) |

#### Пример: разделение ответственности в Laravel

```php
// OrderService — только бизнес-логика заказа
class OrderService
{
    public function placeOrder(array $data): Order
    {
        $order = Order::create($data);
        event(new OrderCreated($order)); // не знает про email и склад
        return $order;
    }
}

// Слушатели подключаются отдельно — легко добавить/убрать
```

## 3. Плюсы и минусы — вопрос на понимание

На экзамене важно показать, что система событий — **инструмент**, а не серебряная пуля. Нужно понимать **когда применять** и **какие риски** несёт.

---

### Плюсы

| Плюс | Объяснение | Пример |
|------|------------|--------|
| **Слабая связанность** | Модуль A не импортирует и не вызывает напрямую B, C, D | Заказ не знает про 10 подписчиков |
| **Расширяемость (Open/Closed)** | Новый функционал — новый listener, без правки ядра | Плагин WordPress |
| **Переиспользование** | Один и тот же хук — много обработчиков | `save_post` → SEO, кэш, уведомления |
| **Тестируемость** | Ядро тестируется отдельно; слушатели мокаются | Unit-тест сервиса без реальной почты |
| **Параллельная разработка** | Команды пишут плагины/модули независимо | Тема + 5 плагинов |
| **Асинхронность** | Тяжёлые задачи уходят в очередь | Отправка email через queue |
| **Гибкость CMS** | Экосистема плагинов — основа WordPress | Тысячи плагинов без патча core |

---

### Минусы и риски

| Минус | Объяснение | Что может пойти не так |
|-------|------------|------------------------|
| **Сложность отладки** | Поток выполнения не линейный | «Кто изменил `$title`?» — 15 фильтров |
| **Неявные зависимости** | Связь только по имени хука | Опечатка в `'init'` vs `'inti'` — тихий баг |
| **Порядок выполнения** | Зависит от priority | Два плагина с приоритетом 10 — непредсказуемо |
| **Производительность** | Много обработчиков в одном запросе | 50 `add_action` на `init` замедляют сайт |
| **«Спагетти» из хуков** | Злоупотребление глобальными хуками | Невозможно проследить цепочку |
| **Сложность тестирования интеграции** | Нужно поднимать dispatcher / WP bootstrap | E2E тяжелее unit |
| **Отсутствие контракта** | Строковые имена без типов | Рефакторинг ломает плагины |
| **Синхронные ошибки** | Падение listener роняет весь запрос | Exception в плагине → 500 на сайте |
| **Безопасность** | Любой код на хуке выполняется в том же контексте | Вредоносный плагин на `admin_init` |

---

### Когда уместно / когда нет

| ✅ Уместно | ❌ Лучше без событий |
|-----------|---------------------|
| Плагинная архитектура (CMS) | Простой CRUD из 3 файлов |
| Много независимых реакций на одно действие | Одна чёткая цепочка из 2 шагов |
| Интеграции (почта, аналитика, webhooks) | Критичная транзакция «всё или ничего» без саги |
| Расширение продукта третьими лицами | Жёсткие инварианты, где порядок критичен и фиксирован |

---

### Как снижать минусы (показать понимание на экзамене)

1. **Именованные классы событий** вместо строк (`OrderCreated`, не `'order_created'`).
2. **Документирование хуков** — какие аргументы, что можно менять.
3. **Разумный priority** и избегание скрытых зависимостей между плагинами.
4. **Очереди** для тяжёлой работы (email, API).
5. **try/catch** вокруг listener или изоляция ошибок (Laravel `ShouldHandleExceptionsAfterCommit`).
6. **Логирование** диспетчеризации в dev-режиме.
7. **Не злоупотреблять filters** там, где нужен явный сервис.

---

### Ответ «на отличие» (шаблон для устного экзамена)

> Система событий и хуков позволяет ядру приложения **публиковать факты** (события) и **не знать** о всех последствиях. Подписчики подключаются через хуки или listeners. Это даёт **расширяемость и слабую связанность**, что критично для CMS вроде WordPress и модульных фреймворков. Цена — **менее очевидный поток выполнения**, риски с **порядком и производительностью**, сложнее **отладка**. Поэтому для простой линейной логики достаточно прямого вызова сервисов, а события оправданы при множественных независимых реакциях и plugin-экосистеме.

---

### Сравнение с прямыми вызовами

```php
// Прямой вызов — жёсткая связь
class OrderService
{
    public function __construct(
        private Mailer $mailer,
        private StockService $stock,
        private Analytics $analytics,
    ) {}

    public function create(array $data): Order
    {
        $order = Order::create($data);
        $this->mailer->send($order);
        $this->stock->decrease($order);
        $this->analytics->track($order);
        return $order;
    }
}

// События — слабая связь
class OrderService
{
    public function create(array $data): Order
    {
        $order = Order::create($data);
        event(new OrderCreated($order));
        return $order;
    }
}
```

| Критерий | Прямой вызов | События |
|----------|--------------|---------|
| Связность | Высокая | Низкая |
| Добавить функцию | Правка `OrderService` | Новый listener |
| Понятность потока | Выше | Ниже |
| Подходит для | Маленьких проектов | CMS, крупных модульных систем |

## Шпаргалка

| Термин | Запомнить |
|--------|-----------|
| **Event** | Что произошло |
| **Hook** | Точка подключения в ядре |
| **Listener / Subscriber** | Кто обрабатывает |
| **Dispatcher** | Кто рассылает |
| **Action** | Сделать что-то (без return) |
| **Filter** | Изменить значение и вернуть |
| **Priority** | Порядок выполнения обработчиков |

### WordPress за 4 функции

```
add_action  + do_action      → события
add_filter  + apply_filters  → преобразование данных
```

### Laravel за 3 сущности

```
Event class + Listener + event() / EventServiceProvider
```
